Factory Girl

Summary

Writing tests allows a developer to create the functionality and test it to make sure it works as predicted. The tests will reveal aspects of the many relationships, and how things are tied together.

A rails developer show do all he can to avoid looking at the browser. This is a weird notion because one can think that being a web developer means you need to view the web.

However, the tools you work in are abstracted so much that they do not need a browser for the internalls. Although, when it becomes time to drive out the integration tests - testing that the user can interact with the web, a browser can be helpful.

The goal though, is to be able to test drive features without ever needing to lean on the web.

There are a few parts

First, setup the file that you will be defining a factory with;

FactoryGirl.define do
end

Then, designate a factory with the model and the name of the ruby model that needs a factory

factory :contact do
end

Add attributes you need inside the factory method block

factory :contact do
  firstname "Jon"
  lastname "smith"
end

Note that the attributes are called as methods, not symbols.

firstname 'jon'

Factory Model

# Make this a factory file;
FactoryGirl.define do

  # Make a factory for the model 'rate_layer'
  factory :rate_layer do

    # What is this;
    factory :seasonal_rates do

      # What is this method?;
      static_rate
      # A: That's an association

      # What is this?;
      name 'seasonal'
      # A: This is the attribute

      # What is this?;
      sort_order 0

      # What are the bloc variables for?
      after(:create) do |rate_layer, evaluator|

        # Where is winter created?;
        FactoryGirl.create(:winter, rate_layer: rate_layer)
      end

    end
  end
end

Shorten the Syntax

Shorten the syntax by adding a configuration to the spec_helper

# spec/spec_helper.rb
RSPec.configure do |config|
  # Include Factory Girl syntax to simplify calls to factories
  config.include FactoryGirl::Syntax::Methods

  # ...

end
