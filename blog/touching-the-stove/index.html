Learning to Code

Touching the stove.

Dec 22, 2014

The whole 'your kids will only touch a hot stove once.' are the kind of lessons I look for with personal coding projects. I want to burn my hand on something relatively harmless. I've rebuilt this site at least 3 times because of I didn't incorporate some fundamentals in earlier versions.

The files were not properly organized.

Things became harder to change after only a few days into the project. That's the first smell. If I have trouble knowing where and how content is organized now, two weeks from now I won't have any idea how this site works.

I've had that happen where I take a break from a project and 2 weeks later I have forgotten most of what I did. I'm going to question my own thought process and notes. So the first few times I built the site, I built it for the external use. I wanted to see my vision look good.

Once I achieve that, I wanted to make some changes. So in adding content to my blog I found use to be a bit clunky.

So, my latest iteration is to organize the code in such a way that it makes sense for my future self.

Lesson 1: Keep files organized.

I duplicated code.

Nothing creates code fatigue quickler than having to repeat my efforts more than once. While building the the site, I couldn't tell if and when I needed something again. But soon I found myself repeating code a third and even fourth time. What you want to do is fix it once and it works everywhere.

Lesson learned: Practice DRY (Don't Repeat Yourself) or you will cry.

I named things poorly.

This is another version of removing duplication. Much of my duplication was due to my naming essentially identical pieces of code, differently. Once I saw the pattern what I needed to do was abstract the naming to something more common. That made it easier to reuse code and remove duplication. win/win

Lesson 2: Name things as generic as possible.

Pages didn't match

One of the consequences of sloppy code is inconsistency. Headers are too big or too small and there are little hacks all around the stylesheets to adjust. These only make for future fatigue.

Lesson 3: Sloppy code makes pages inconsistent.

Users don't like me as much as I like myself

My first version had a really cool bird I designed. I was proud of it and thought it made for a cool landing page. But I've realized that landing pages are not user-friendly. The clock starts when I land on your page and you've got millaseconds to convince me to stay and click something else.

Lesson 4: Make the site User Friendly.

The content on my site was buried. So I completly removed the page and opted for a timeline format. That way you can see the latest content and determine if it's worth clicking further. And since I go to the trouble of adding content, I hope that you find something you want.

Takeaway

These are fundamentals I needed to know as a developer. I had a sense that these are the kind of topics you learn by not adhering to them. These are things to think about when it comes to estimating a project. If you plan to actually use the software you write, then you will need to take the time to make it right.
